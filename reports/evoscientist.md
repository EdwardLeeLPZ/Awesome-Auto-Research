## 2. Architecture

EvoScientist adopts a **multi-agent team architecture** with specialized sub-agents coordinated through a central research lifecycle engine.

### 2.1 Core Multi-Agent Team

**Six Specialized Sub-Agents:**

1. **Planner Agent**
   - Role: Decompose high-level research questions into actionable steps
   - Output: Structured research plan with dependencies and milestones
   - Decision: When to branch, when to go deep, when to pivot

2. **Research Agent**
   - Role: Conduct literature review and knowledge synthesis
   - Capabilities: Multi-source literature search, paper summarization, knowledge graph construction
   - Integration: Connects to PubMed, ArXiv, BioRxiv, general web

3. **Code Execution Agent**
   - Role: Write and execute experimental code
   - Sandbox: Docker-based or local Python interpreter with resource limits
   - Feedback: Capture stdout, stderr, execution time, intermediate results

4. **Debug Agent**
   - Role: Diagnose and recover from failures
   - Strategy: Error analysis, hyperparameter tuning, alternative approaches
   - Persistence: Maintains experiment log for pattern recognition

5. **Analysis Agent**
   - Role: Statistical analysis and result interpretation
   - Methods: Significance testing, visualization generation, trend analysis
   - Output: Structured findings with confidence levels and caveats

6. **Writing Agent**
   - Role: Compose research manuscript, abstract, supporting materials
   - Format: LaTeX/Markdown with figures, tables, mathematical notation
   - Compliance: Follows journal submission guidelines

### 2.2 Persistent Memory System

**Context Preservation:**
- **Session Memory:** Working context, intermediate findings, experiment results persists across agent calls
- **Preference Learning:** System learns user's research taste, writing style, preferred methodologies
- **Experiment Registry:** Searchable log of all experiments with metadata (date, parameters, outcomes)
- **Knowledge Base:** Accumulated insights that inform future planning and hypothesis generation

### 2.3 Unified Control Layer & Multi-Channel Interface

**Multiple Deployment Surfaces (Single Backend):**

| Channel | Interface | Use Case |
|---------|-----------|----------|
| CLI/TUI | Terminal-based | Local development, batch workflows |
| Web UI | Gradio/Streamlit | Browser-based monitoring |
| Telegram | Chat bot | Mobile notifications, quick prompts |
| Slack | Integration | Team collaboration |
| Feishu | Enterprise | Corporate research teams |
| WeChat | Chinese users | Community support |

**Unified backend ensures:**
- Single research session across all interfaces
- Consistent state synchronization
- Role-based access control

### 2.4 Scientific Workflow Engine

**Research Pipeline Phases:**

```
INTAKE → PLAN → EXECUTE → EVALUATE → WRITE → VERIFY → ITERATE
  ↓       ↓      ↓        ↓         ↓      ↓       ↑
User  Decompose Code+    Analysis  Compose Peer    Feedback
Input Methods  Expts     Results   Draft   Review  Loop
```

Each phase produces artifacts (plans, code, results, drafts) that feed into the next phase.

### 2.5 MCP & Skills Extensibility

**Pluggable Components:**
- MCP servers for integrating external tools (BioQuery, ChemicalShift, etc.)
- GitHub-hosted skills for domain-specific research workflows
- One-command installation: `evo skill install <skill-url>`

## 3. Core Workflow

### 3.1 Standard Research Session

**Phase 1: Intake & Planning (Planner Agent)**
- User: "Investigate the role of protein X in cancer metastasis"
- System: Decomposes into sub-questions
  - What is known about protein X in the literature?
  - What experimental methods are standard?
  - What data is publicly available?
  - What hypotheses should we test?

**Phase 2: Literature Research (Research Agent)**
- Parallel search: PubMed, ArXiv, BioRxiv for "protein X cancer"
- Summarization: Extract key findings, identify gaps
- Output: Literature summary + gap analysis + recommended experiments

**Phase 3: Experiment Design & Execution (Code & Debug Agents)**
- Code agent writes Python scripts for data analysis or simulation
- Execution: Runs in sandboxed environment with resource limits
- Failure handling: Debug agent diagnoses errors, suggests fixes
- Iteration: Multiple experiment rounds with parameter tuning

**Phase 4: Analysis & Interpretation (Analysis Agent)**
- Statistical testing on experimental results
- Visualization generation (plots, heatmaps, network diagrams)
- Structured findings: "Result X shows significant difference (p<0.05)"

**Phase 5: Manuscript Generation (Writing Agent)**
- Compose abstract, introduction, methods, results, discussion
- Incorporate figures and tables from analysis phase
- Validate: Cross-check claims against experimental results

**Phase 6: Peer Review & Verification (Review Loop)**
- LLM-based reviewer provides critique (scientific rigor, clarity, novelty)
- Author agent refines manuscript based on feedback
- Multiple review iterations until quality threshold reached

### 3.2 Self-Evolution Phase

**Reinforcement Learning Training (Offline):**
- Each completed research session generates training data:
  - State: research question, current findings, agent decisions
  - Action: which method/tool agent chose
  - Reward: paper acceptance signal, peer review score, citation potential
- RL algorithms (PPO, GRPO, or similar) fine-tune agent policies
- Next iteration: Agents make better decisions based on learned preferences

### 3.3 Human-on-the-Loop Integration

**Rather than fully autonomous**, EvoScientist supports:
- **Interactive checkpoints:** "Before running expensive experiments, should we proceed?"
- **Preference feedback:** "I prefer statistical modeling over simulation for this"
- **Guidance injection:** "Try this specific dataset" or "Avoid this method"
- **Co-authorship paradigm:** AI generates draft, human refines for submission

## 4. Key Features

### 4.1 Multi-Agent Team with Specialization
Six agents with distinct roles eliminate the need for monolithic LLM chains. Each agent is fine-tuned for its specialty, improving coherence and reducing hallucinations.

### 4.2 Deep Research Capability
- Recursive depth/breadth exploration of research directions
- Automatic branch pruning (low-confidence paths) and depth-first exploration of high-confidence paths
- DeepResearch Bench II ranking #1 validates capability

### 4.3 Persistent Memory & Learning
- Session context survives across restarts
- User preferences and research taste internalized over time
- Experiment history enables pattern recognition and meta-learning

### 4.4 Multi-Provider LLM Support
- Anthropic (Claude 3.5, Claude 4)
- OpenAI (GPT-4o)
- Google (Gemini)
- MiniMax
- NVIDIA
- **Benefit:** Flexibility and cost optimization; switch providers based on task complexity

### 4.5 Multi-Channel Deployment
- Single backend synchronized across CLI, web, Telegram, Slack, WeChat, Feishu
- Ideal for research teams with diverse communication preferences
- Mobile-friendly for researchers on-the-go

### 4.6 MCP Integration
- Plug in custom MCP servers on-the-fly
- Install GitHub skills with one command
- Extensible without code changes

### 4.7 Scientific Workflow Lifecycle
- Well-defined phases (INTAKE → PLAN → EXECUTE → EVALUATE → WRITE → VERIFY)
- Each phase produces validated artifacts
- Automatic orchestration with fallback mechanisms

### 4.8 RL-Based Self-Improvement
- Learns from every completed research session
- Refines policies for future planning, method selection, and writing
- Enables progressive quality improvement without manual tuning

## 5. Technical Implementation

### 5.1 Backend Stack
- **Language:** Python 3.11–3.13 (excludes 3.14 due to breaking changes)
- **Core Framework:** LangChain / LangGraph for agent orchestration
- **Sandbox:** Docker containers or local `subprocess` with resource limits
- **Async Runtime:** asyncio for non-blocking I/O (API calls, long-running experiments)
- **State Management:** In-memory cache + persistent SQLite for sessions
- **RL Framework:** (Inferred) PPO, GRPO, or A3C for policy training

### 5.2 Agent Orchestration
- Each agent is a LangGraph node with:
  - Input schema (expected state)
  - LLM system prompt (role-specific instructions)
  - Output parser (structured extraction)
  - Error handler (fallback logic)
- Central orchestrator: Directs flow based on research phase and agent outputs
- Inter-agent communication: State graph passed between agents

### 5.3 Code Execution Sandbox
- **Container-based:** Docker images pre-loaded with scientific libraries
- **Local fallback:** Subprocess with whitelist/blacklist for imports
- **Resource limits:** CPU time, memory, network bandwidth constraints
- **Output capture:** Stdout, stderr, file artifacts collected for analysis

### 5.4 LLM Configuration
- **Environment variables:** `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, etc.
- **YAML config:** Specify model, temperature, max_tokens per agent
- **Dynamic routing:** Select provider based on task complexity or cost budget

### 5.5 Multi-Channel Implementation
- **CLI:** Rich TUI using `typer` + `rich` libraries
- **Web UI:** Gradio or Streamlit for browser-based interface
- **Chat APIs:** Telegram Bot API, Slack SDK, WeChat enterprise API
- **Message Queue:** Redis or similar for state synchronization across channels

## 8. Limitations

1. **Early-Stage Release**
   - Project launched publicly only in March 2026 (very recent)
   - Long-term stability and corner-case handling unproven
   - PyPI version 0.0.2 indicates active development and potential breaking changes

2. **Computational Cost**
   - Six parallel agents + RL training implies significant LLM token usage
   - Each research session may incur hundreds of API calls
   - Cost control mechanisms not documented (no usage estimates provided)

3. **Execution Latency**
   - Multi-phase workflow introduces sequential bottlenecks
   - Approval checkpoints in human-on-the-loop mode delay progress
   - Sandbox startup (Docker) adds overhead per experiment

4. **RL Training Complexity**
   - Self-evolution requires sufficient data volume (many completed sessions)
   - Sparse reward signal (acceptance/rejection) may slow learning convergence
   - Hyperparameter tuning for RL algorithms not documented

5. **Limited Domain Coverage**
   - ICAIS 2025 papers suggest AI/ML research focus
   - Generalization to biology, chemistry, physics not yet demonstrated
   - Skill ecosystem (EvoSkills) still developing

6. **Documentation Gaps**
   - Technical report is recent (11 Mar 2026) but may not cover all features
   - Multi-channel setup (Telegram, Slack, Feishu) configuration minimally documented
   - Troubleshooting and debugging guidance sparse

7. **Sandbox Security**
   - Code execution from untrusted prompts poses injection risks
   - Docker/subprocess isolation is defense-in-depth but not foolproof
   - Resource limits (CPU, memory) may be circumvented by sophisticated attacks

8. **LLM Dependency**
   - Workflow quality tied to underlying model capability
   - Weaker models may fail at planning or code generation stages
   - Multi-phase design doesn't eliminate hallucination risk entirely

### Comparison with Other End-to-End Systems

| System | Multi-Agent | RL Training | Memory | Multi-Channel | Focus |
|--------|-------------|-----------|--------|---------------|-------|
| **EvoScientist** | ✅ (6 agents) | ✅ Yes | ✅ Persistent | ✅ (6+ channels) | Full research lifecycle |
| **AI-Scientist** | ⚠️ Basic | ❌ No | ❌ No | CLI only | ML research template-driven |
| **AI-Scientist-v2** | ✅ (BFTS tree search) | ❌ No | ❌ No | CLI only | ML research with peer review |
| **AI-Researcher** | ⚠️ Basic | ❌ No | ⚠️ Limited | CLI only | Full lifecycle, reproducibility-focused |
| **Agent Laboratory** | ✅ (role-based) | ❌ No | ❌ No | CLI only | Role-specialized architecture |

### Shared Design Patterns
- **Multi-agent orchestration:** Mirrors Agent Laboratory and AI-Scientist-v2; EvoScientist adds RL refinement
- **Peer review integration:** Like AI-Scientist-v2; EvoScientist uses this for reward signal
- **Persistent state:** Similar to sophisticated coding agents (e.g., Aider); EvoScientist extends to research context
- **LLM provider abstraction:** Common pattern; EvoScientist supports 6+ providers out of the box

### Differentiation
EvoScientist is uniquely positioned for **long-running research partnerships** with self-improvement. The RL mechanism and human-on-the-loop paradigm distinguish it from prior end-to-end systems.
